// How this tool works

From part number to specs

The pipeline

Every part number gets routed by its prefix (M3/M4 → Samsung, HMC → SK Hynix DDR5, HMA/HMT → SK Hynix DDR4/DDR3, MT → Micron, KSM/KVR/KF → Kingston, CT/BL → Crucial). Each manufacturer has its own decoder built from their official part-number cards.

The decoder slices the PN at fixed character positions and looks each segment up in a table. What you see in the Segment Breakdown grid is one entry per slice — that's the raw character (or pair) and what it stands for.

How segments are decoded

Most fields are simple character-position lookups. For example, in HMCG78MEBRA110N:

HMCpositions 1–3 → "DDR5 Module" G7positions 4–5 → 16GB module density 8position 6 → x8 bit organization Mposition 7 → 1st-generation die EBpositions 8–9 → DDR5-4800 CL40-39-39 Rposition 10 → RDIMM Aposition 11 → 16Gb die, low profile, commercial temp 110positions 12–14 → Rambus RCD + TI 5010 PMIC Nposition 15 → Pb-free / RoHS

How specs get built

The Specifications grid is the segment data reorganized into normal vendor-spec language — capacity, speed, module type, ECC, voltage, etc. It mostly mirrors the segments, but a few fields combine multiple positions:

Capacity comes directly from the density code on DDR5 (G7=16GB). On DDR4 it's computed from the depth code times the data bus width.

ECC, Registered, Load Reduced, Pin Count are derived from the module-type table for that family — RDIMM implies ECC + Registered + 288-pin, SODIMM implies non-ECC + 260/262-pin, etc.

Voltage is set from the DDR generation: 1.5V for DDR3, 1.35V for DDR3L, 1.2V for DDR4, 1.1V for DDR5.

How rank is calculated

This is the tricky one. Most server PNs (Samsung M3xx, Hynix HMA/HMC, Micron MT) don't encode rank directly — it has to be derived. The math is:

ranks = capacity_GB ÷ (chips_per_rank × die_GB)

where chips_per_rank = 64 ÷ chip_width_bits on a 64-bit data bus (ECC parity chips don't add capacity). Die density is read from the schema for DDR5, or inferred from the depth-code suffix and component-revision letter for DDR4.

Kingston KSM and KVR encode rank as a single character in the PN (S=1R, D=2R, Q=4R, O=8R). Those decode straight from the character — no math, no inference.

The badge next to each rank tells you which path it took: Encoded = read from PN, Est. = calculated, Verify = not in PN at all.

What it doesn't know

SPD-only details (XMP profiles, exact tFAW/tRRD, lot codes, factory revision dates) aren't in the part number. Brand-specific cosmetic data (heatspreader color, RGB) isn't either. Anything Crucial, FURY, G.Skill, Corsair, or TeamGroup keeps proprietary won't decode beyond capacity and speed.
